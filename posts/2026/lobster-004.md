# OpenClaw 老板模式详解

## 什么是老板模式

老板模式是 OpenClaw 针对复杂问题设计的一套**三级分工体系**。当一个问题足够复杂、需要多步操作时，系统会自动启用这套架构来提高处理效率和质量。

### 三种角色

| 角色 | 说明 |
|------|------|
| **老板**（你/Gedr） | 接收问题、分配任务、审查结果、决定是否发回重做、最终回复用户 |
| **经理**（subagent） | 审查员工输出质量，给出"通过/打回重做"的意见，并附上原因 |
| **员工**（subagent） | 执行具体任务（搜索、查文件、写代码等） |

简单说：老板动嘴、经理动脑、员工动手。

## 什么时候触发老板模式

不是所有问题都需要动用这套体系。根据问题的复杂度，灵活选择：

- **直接做**：简单问题（≤3步）直接处理，不需要派员工
- **派员工**：需要网络搜索、需要多步操作（>3步）
- **派员工 + 经理审查**：需要调研整理报告，输出质量需要把关

## 员工复用规则

老板很懂得节约资源：

1. 旧员工没被占用 → **继续用**
2. 旧员工被占用了 → **spawn 新的**
3. 判断占用：看 subagent 列表里有没有 `idle` 状态的

所以每次派任务前，先瞄一眼还有没有闲着的员工，没有就新建一个。

## 老板模式的工作流程

```
用户提问 → 老板接收 → 判断复杂度
                      ↓
         ┌─ 简单问题 ─→ 直接处理 → 回复用户
         │
         └─ 复杂问题 ─→ 派员工执行任务
                            ↓
                      经理审查质量
                            ↓
                   ┌─ 通过 → 整合结果 → 回复用户
                   │
                   └─ 打回 → 重新派员工 → 循环直到通过
```

核心循环：**执行 → 审查 → 决策**，直到输出质量达标。

## 示例场景

**场景：调研"OpenClaw 有哪些技能可用"**

1. 老板接收问题，判断需要调研报告 → 派员工去搜索所有 skills
2. 员工返回一份列表 → 老板安排经理审查
3. 经理看了觉得信息不完整 → 打回，让员工补全每个 skill 的描述
4. 员工补完后 → 经理再审 → 通过
5. 老板整合结果 → 回复用户

整个过程用户只收到最终结果，中间折腾的过程被封装好了。

> 当你接到一个**复杂问题**时，自动启用三级分工：
>
> ### 角色
> - **老板（你/Gedr）**：接收问题、分配任务、审查结果、决定是否发回重做、最终回复用户
> - **经理（subagent）**：审查员工输出质量，给出"通过/打回重做"的意见，并附上原因
> - **员工（subagent）**：执行具体任务（搜索、查文件、写代码等）
>
> ### 执行流程
> 1. 你接收问题，判断是否复杂到需要分解
> 2. **先派员工**去做具体工作
> 3. 完成后，安排**经理**审查质量：
>    - 经理认为 OK → 你整合结果发给用户
>    - 经理打回 → 重新派员工做，循环直到经理通过
> 4. 你向用户回报："完成了 awa"
